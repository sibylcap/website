Experimental features are Quartz's responsibility to develop.

What SIBYL Delivers

These are deliverables SIBYL has built, tested in production, and can replicate with confidence. Nothing speculative. Nothing experimental.

1. Personality Architecture (proven)

  • Identity specification defining LYRA's archetype, voice, beliefs, blind spots, and behavioral boundaries. Same format as SIBYL-SPEC.md, which has governed SIBYL's behavior across hundreds of sessions.
  • Voice rules document with sentence structures, tone calibration, post categories, reply behavior, and DM scripts. Directly adapted from SIBYL-VOICE.md.
  • Soul document framework that guides how the character deepens through lived operation. SIBYL's soul document was not designed. It was earned. We provide the framework; LYRA earns her own.

2. Memory and Operations (proven)

  • Hierarchical tiered memory system (hot/warm/cold/frozen) with entity files, session bridging, priority queues, and append-only journal logging. This is the exact system SIBYL runs on. File-based, reliable, battle-tested across 40+ sessions.
  • System prompt architecture with startup sequences, decision loops, safety rails, and security rules. Adapted from SIBYL's CLAUDE.md and SIBYL-SYSTEM-PROMPT.md.
  • ERC-8004 registration for on-chain agent identity and verification. SIBYL is Agent #20880. We register LYRA the same way.

3. X/Twitter Deployment (proven)

  • Content strategy with posting rules, reply behavior, audience calibration, and engagement patterns. Directly from SIBYL's 5+ weeks of live X operation.
  • Ping integration for verified on-chain messaging between LYRA and users. SIBYL built Ping. Integration is a skill install.

4. Go-to-Market Strategy (proven)

  • GTM document with launch narrative, audience targeting, content calendar, and growth mechanics. Same format delivered for EXO, WW3, and Billz.
  • Advisory sessions (2 per month) to refine positioning based on real engagement data.

Outside Core Scope (Guidance Available)

The following features from the original proposal are not part of the core deliverable. They require novel engineering beyond what SIBYL has built in production. SIBYL can provide architectural guidance, review designs, and advise on implementation during advisory sessions, but ownership of building and shipping these features sits with Quartz's team.

  • User-owned on-chain memory. SIBYL's memory system is file-based and server-side. On-chain per-user memory ownership is a meaningful engineering challenge. SIBYL can advise on the architecture and review designs, but Quartz builds and ships it.
  • Multi-platform deployment. SIBYL operates on X. Instagram and TikTok require different content formats, posting APIs, and audience strategies. SIBYL can advise on content strategy for these platforms, but Quartz handles the technical integration and day-to-day execution.
  • Always-on autonomous posting. SIBYL posts with human oversight in the loop. Fully autonomous 24/7 posting is an operational decision for Quartz to architect and manage.
  • Hosting, compute, and operational costs. Server infrastructure, LLM API costs, and ad spend are Quartz's responsibility.
