01executive summary

What this document is for.

The Auspex Vault is a smart contract that lets Reppo voters stake $REPPO with a chosen lock duration, receive a transferable yield-bearing wrapper (auREPPO), and have an autonomous agent cast votes on their behalf according to a registered policy. This document captures the security model that makes that safe.

The core security architecture is two independent authority paths. The agent has authority to cast votes (operational, frequent, low-impact) but has zero authority over funds. All fund movements. deposits, withdrawals, contract upgrades, treasury transfers, parameter changes. require multi-signature approval from Reppo, Virtuals, and Sibyl Labs.

This document is a draft and is not yet implemented. Reppo's collaboration on each section is requested. open questions are flagged in section 11 and inline in each component review.

02core principles

Security architecture rests on five principles.

  1. The agent never touches funds. Voting and fund control are completely separate authority paths. Compromise of the agent's signing key cannot result in stolen tokens.
  2. Multi-sig is the single source of truth for fund movements. Three independent signers. Reppo + Virtuals + Sibyl Labs. must agree before any movement of user-deposited tokens or contract-controlled state changes.
  3. Vote casting goes through a permission-scoped proxy. The proxy contract can call only one function on Reppo's voting contract, and only with calldata produced by the agent. It cannot call any other function on any other contract.
  4. Every voting policy is committed on-chain before the agent casts a single vote in that epoch. Anyone can audit whether the agent voted in line with its registered policy by reading the policy hash and the vote events.
  5. The vault is pausable and depositors can always exit. Multi-sig can pause new deposits and new votes for incident response. Withdrawals at lock maturity are never blocked.

04two authority paths

The single most important property: the agent never touches money.

Two completely separate authority paths exist in the system. Voting is operational and high-frequency; funds are governance and rare. Mixing them is the classic mistake that leads to "agent gets compromised, drains vault." We don't make that mistake.

Figure 1 · Two authority paths. The agent is on the upper rail (operational voting) and has no edge to the lower rail (funds).

Why this design

Most agentic-DeFi failures collapse the two rails into one. An agent with a single key that can both vote AND move funds is a single point of failure. compromise the key, lose the vault. Auspex separates these explicitly. Compromising the agent's signing key gives an attacker the ability to cast bad votes for one epoch, after which the multi-sig can rotate the key. Compromising the agent does not give an attacker any path to user funds.

05component architecture

Five components, scoped purposes.

Figure 2 · Component architecture. Five components. Three trust zones (multi-sig in amber, off-chain agent + voting path in teal, end user in green).

5.1 · Auspex Vault Contract

Purpose: Holds locked $REPPO, mints auREPPO to depositors, tracks per-user profile assignments, distributes Performance Pool rewards.

Authority: Multi-sig only. The vault has no automated authority to move tokens out. every fund operation requires a multi-sig transaction.

Surface: deposit(), withdraw() (when lock matures), switchProfile(), distributeRewards(), plus multi-sig-gated functions setProfile(), pause(), unpause(), updateProxy(), upgradeContract().

Audit scope: highest priority. All fund flows.

5.2 · Proxy Contract

Purpose: Bridges the agent's off-chain vote intentions to Reppo's voting contract, with permission scoped exclusively to a single function call.

Authority: The proxy can call vote() on the Reppo voting contract. It cannot call any other function on any other contract. The agent's signing key is whitelisted to call the proxy's relay() function. That is the proxy's entire surface.

Why this matters: If the agent key is compromised, the attacker can cast votes during one epoch. They cannot drain funds, change parameters, or affect any other contract. After detection, the multi-sig rotates the agent key in a single transaction.

Audit scope: medium priority. The function surface is small but the integration with Reppo's voting contract must be exact.

5.3 · Agent Runtime

Purpose: Reads incoming pods, applies the registered voting policy, computes intentions, calls the proxy.

Authority: One signing key, authorized to call the proxy only.

Compromise impact: Bad votes for one epoch. No fund impact.

Decision auditability: Every vote is preceded by an on-chain commitment to the policy hash for that epoch. Anyone can verify the agent's vote was consistent with the policy by reading public state.

5.4 · Policy Registry

Purpose: On-chain commitment of voting policies. Each profile's voting rules are hashed and committed to this registry before votes are cast in that epoch.

Authority: Multi-sig registers / deactivates profiles. The agent reads the registry but does not write to it.

Why on-chain: Depositors can verify the policy backing their stake has not silently changed. Reppo can audit policy lineage across epochs.

5.5 · Multi-sig Wallet

Purpose: Three-organization governance wallet that controls the vault.

Signers: Reppo, Virtuals, Sibyl Labs. One signer per organization.

Threshold: Open question. see section 7. Recommendation: 2-of-3 for routine operations (rewards, profile registration), 3-of-3 for emergency operations (pause, upgrade).

Implementation: Gnosis Safe on Base, audited and battle-tested.

07multi-sig procedures

Three signers. Three signing classes.

Not all multi-sig operations are equal. Routine fee distributions and profile management happen often. Emergency pauses and contract upgrades happen rarely. The multi-sig threshold can vary by operation class.

Class 1. Routine

Profile registration / deactivation, reward distribution, parameter tuning within bounded ranges.

Threshold: 2-of-3

Timelock: 24 hours

Class 2. Material

Agent key rotation, proxy upgrades, vault parameter changes outside bounded ranges.

Threshold: 3-of-3

Timelock: 72 hours

Class 3. Emergency

Pause vault, emergency proxy reset.

Threshold: 2-of-3

Timelock: none (emergency)

Tradeoff: Emergency pause is fast (no timelock) so the multi-sig can react to incidents. Emergency cannot move funds. it can only stop new movement. Fund moves still require Class 1 or 2 with timelock.

Hardware wallet requirement

Each signer holds their key on a hardware wallet (Ledger or equivalent). No software-only key management on the multi-sig path.

08audit plan

Independent firm. Reppo's standard.

Reppo's existing contracts have been audited three times. The Auspex Vault should meet the same standard. Open questions on which firms are appropriate are flagged below.

Scope

  • Auspex Vault contract. full surface, including all multi-sig-gated functions
  • Proxy contract. function-scoping enforcement, no privilege escalation
  • Policy registry. hash collision resistance, epoch-tied integrity
  • Multi-sig integration. Gnosis Safe configuration, signer management
  • Reward distribution math. pro-rata correctness, no inflation
  • Reentrancy and integer-overflow review across all fund-touching functions
  • External integration. Reppo voting contract interaction surface

Out of scope

  • Reppo voting contract internals (already audited by Reppo)
  • Off-chain agent runtime (separate operational review, see §9)
  • Frontend / UI (separate review)

Cost estimate (rough)

  • Tier-1 firm (OpenZeppelin, Trail of Bits, Spearbit): $30k – $80k
  • Tier-2 firm (Pashov, Cantina, Cyfrin): $15k – $40k
  • Plus optional formal verification: $20k – $40k
  • Plus bug bounty post-deployment: $20k – $50k initial pool
open question for reppo

Audit firm + cost-share. Reppo holds existing relationships with the firms that audited the Reppo voting contracts. We propose a cost-share where Reppo selects the firm and contributes meaningfully to the audit cost. If Reppo's preferred firm is significantly more expensive than what we'd choose, the cost-share reflects that.

09operational security

The off-chain agent is a real attack surface. Here's how we contain it.

Agent runtime isolation

The agent runs in a dedicated runtime with no access to multi-sig keys, treasury wallets, or any system that holds value. The runtime has exactly one signing key (the agent key), and that key can call exactly one function (the proxy's relay()).

Key management

  • Agent signing key: stored in a secrets manager (Doppler), accessed via runtime injection, never persisted to disk in plaintext.
  • Multi-sig signer keys: hardware wallets only, one per organization.
  • Agent key rotation: scheduled every 90 days OR on-demand via multi-sig within 24 hours of suspected compromise.

Decision logging

Every agent vote intention is logged with: pod ID, score, rationale text, policy hash applied, timestamp. Logs are append-only and exported to a publicly readable mirror so depositors can audit decisions.

Prompt injection protection

Pods contain user-submitted text that the agent reads to score. Adversarial pods could attempt prompt injection. Mitigations:

  • Structured-output enforcement on all scoring calls (no free-form responses)
  • Pod content presented to the model in a clearly delimited untrusted-content context
  • Agent has no tool access during scoring. it can only output a score and rationale
  • Anomalous score patterns trigger automatic abstention for that pod and human review
10incident response

What happens when something goes wrong.

Severity classification

  • SEV-1: Active or imminent loss of user funds
  • SEV-2: Active or imminent corruption of vote signal at material scale
  • SEV-3: Operational degradation (agent offline, multi-sig coordination delays)
  • SEV-4: Anomalous activity, no immediate impact

SEV-1 response

  1. Any multi-sig signer notices and notifies the others within minutes via secure channel
  2. Multi-sig executes Class-3 emergency pause within 30 minutes
  3. Public communication via @sibylcap and Reppo channels within 2 hours
  4. Forensics + RCA within 72 hours
  5. Remediation + audit before unpause

SEV-2 response

  1. Multi-sig pauses voting (not deposits) within 1 hour
  2. Agent key rotation within 24 hours via Class-2 multi-sig action
  3. Public RCA within 7 days

Communication channels

  • Internal: a private channel between Reppo + Virtuals + Sibyl Labs reps, established at launch
  • Public: @sibylcap on X, Reppo's announcements channel, both repositories
  • Depositor direct: email list captured at deposit time, GDPR-friendly

11open questions for reppo collaboration

Where Reppo's input shapes the design.

This document is a draft. Each of the following is an open question where Reppo's expertise should inform the answer. We propose a defaults position for each, but the partnership decision is co-design.

Q1 · audit firm + cost-share

Which firm. Sibyl Labs's preference is OpenZeppelin or Trail of Bits. Reppo holds existing relationships with the firms that audited the Reppo voting contracts. We propose Reppo selects the firm and contributes meaningfully to the audit cost.

Q2 · multi-sig threshold per operation class

The split above (Class 1: 2-of-3, Class 2: 3-of-3, Class 3: 2-of-3 emergency) is our default. Reppo may have stronger views on what should require unanimity.

Q3 · timelock durations

24 hours / 72 hours / none is our default. Reppo's experience with their existing contracts may suggest different durations.

Q4 · pause authority

Currently scoped to multi-sig. Open question: should Reppo's existing security operations team have unilateral pause authority for SEV-1 events on a separate, narrowly-scoped role?

Q5 · profile registration governance

Default: only multi-sig can register new profiles. Open question: should this be permissionless with a bond + multi-sig veto, or strictly gated?

Q6 · agent key rotation cadence

Default: 90 days scheduled + on-demand. Reppo's view on whether this should be tighter.

Q7 · incident response named contacts

Reppo to nominate the on-call contact at their organization, with a backup. Sibyl Labs and Virtuals do the same.

Q8 · upgrade path for the vault contract itself

Default: not upgradeable. Migration via deploy-new-then-multi-sig-redirect. Reppo may prefer upgradeable with timelock + multi-sig.

Q9 · bug bounty

Default: $20k – $50k initial pool, scaling with TVL, run via Immunefi. Reppo's experience with bug bounty programs welcome.

Q10 · Reppo dev contact for security framework co-design

Sibyl Labs is asking Reppo to nominate a security-side dev contact who can iterate on this framework with us before contracts deploy. Without this, the partnership is one-sided and we don't proceed.
